# 채팅 시스템 개발 하네스 오케스트레이터

9개 전문 에이전트를 파이프라인 + 3중 생성-검증 루프(계획·개발·PR)로 조율해, 사용자의 개발 요청을 계획부터 PR·설명·완료감사까지 완결한다.

**실행 모드: 에이전트 팀.** 개발↔검수, PR작성↔PR점검 사이의 피드백 루프가 있어 팀 통신이 유리하다.

## Phase 0: 컨텍스트 확인

시작 시 실행 모드를 판별한다:
- `_workspace/` 존재 + 부분 수정 요청 → **부분 재실행** (해당 에이전트만)
- `_workspace/` 존재 + 새 입력 → **새 실행** (기존을 `_workspace_prev/`로 이동)
- `_workspace/` 미존재 → **초기 실행**

## Phase 1: 명료화 게이트 (clarifier)

**항상 먼저** `clarifier`로 요청의 모호점·비가역 작업·숨은 가정을 점검한다.
- 판정 "확인 필요" → 정리된 질문을 **AskUserQuestion으로 사용자에게 직접** 묻는다(clarifier는 질문을 만들 뿐, 실제 묻기는 오케스트레이터가). 답을 받고 진행.
- 판정 "명확" → Phase 2로.

> 이 게이트가 이 하네스의 1번 가치다. "확실하지 않으면 추측 말고 물어본다"를 파이프라인이 강제한다.

## Phase 2: 계획↔검증 루프 (planner ⇄ plan-reviewer)

1. `planner`로 구현 계획서를 만든다. 영향 범위·PR 단위 단계·리스크·테스트 계획. `_workspace/01_plan.md`에 저장.
2. `plan-reviewer`가 완결성·영향범위 정확성·PR 단위·리스크 누락·테스트 구체성·아키텍처 정합을 검증.
3. 보류(차단)면 planner로 되돌려 수정 → 재검증. **승인까지 루프.**

> 계획에도 generate-verify를 두는 이유: 경계면을 놓친 그럴듯한 계획으로 개발을 시작했다 검수 단계에서 뒤엎는 게 가장 흔한 낭비다. 가장 싼 지점에서 계획 결함을 잡는다.

## Phase 3: 개발↔검수 루프 (developer ⇄ reviewer)

팀을 만들어 두 에이전트가 조율한다:
1. `developer`가 계획대로 구현, `pnpm typecheck`+`pnpm test` 통과시킴.
2. `reviewer`가 경계면 정합성·실패 시나리오·아키텍처 위반·테스트를 검수.
3. 보류(CRITICAL/HIGH)면 developer로 되돌려 수정 → 재검수. **통과까지 루프.**
4. 브라우저 관찰 대상이면 실제 동작 확인까지.

개발 중 새 모호점이 나오면 clarifier로 에스컬레이션.

## Phase 4: PR 작성↔점검 루프 (pr-author ⇄ pr-reviewer)

1. `pr-author`가 정책대로 브랜치→PR 생성 (master에서 새 브랜치, 한 PR=한 변경, Conventional Commits).
2. `pr-reviewer`가 base·파일범위·완결성·정합성·롤백성 점검.
3. 변경요청이면 되돌려 수정 → 재점검. 승인까지 루프.

## Phase 5: 변경 설명 (change-explainer)

`change-explainer`가 실제 diff 기반으로 사용자용 한국어 설명을 만든다.

## Phase 6: 완료 감사 게이트 (completion-auditor)

사용자에게 "완료"라고 보고하기 **직전**, `completion-auditor`가 모든 완료 주장을 1차 증거로 재검증한다 — 파일 실존·커밋 존재(push≠커밋)·PR 상태·테스트 실제 실행·실제 동작 관찰. 반증이 하나라도 나오면 "완료"라 말하지 않고, 무엇이 실제로 됐고 안 됐는지 정확히 보고한 뒤 해당 주체로 되돌린다.

> 이 프로젝트에서 가장 자주 겪은 문제가 "됐다는데 안 된 것"이다. 이 게이트가 최종 안전망이다.

## Phase 7: 머지 (사용자 승인)

**머지는 사용자 승인 후에만.** 승인 시 `gh pr merge --squash --delete-branch`. 머지 후 로컬 master 갱신 + typecheck/test 재확인.

## 데이터 전달

- **태스크 기반**(TaskCreate): 진행상황·의존관계.
- **파일 기반**(`_workspace/{phase}_{agent}_{artifact}`): 계획서·검수 결과 등 산출물. 최종물만 사용자 경로/PR로, 중간 파일은 보존.
- **메시지 기반**(SendMessage): 루프 중 피드백.

## 에러 핸들링

- 에이전트 1회 재시도 후 재실패 시, 결과 없이 진행하되 보고서에 누락 명시.
- 상충 데이터는 삭제하지 말고 출처 병기.
- 파괴적 작업(force push, 브랜치 삭제, 데이터 덮어쓰기)은 실행 전 대상·롤백법 명시하고 필요 시 사용자 확인.

## 테스트 시나리오

**정상 흐름**: "soop도 이미지 캡처되게 해줘" → clarifier(chzzk만? 호버/클릭? 확인) → planner(계획) → developer 구현 + reviewer 검수 통과 → pr-author PR → pr-reviewer 승인 → change-explainer 설명 → 사용자 승인 후 머지.

**에러 흐름**: reviewer가 "soop만 연결 시 providerCounts.chzzk undefined로 프레임 안 뜸" CRITICAL 발견 → developer로 되돌림 → fallback 로직 추가 + 회귀 테스트 → 재검수 통과 → 이후 정상 진행.

## 파이프라인 밖: 상시 자문 에이전트

`architecture-guide`는 개발 흐름에 속하지 않는다. 사용자가 "이거 왜 이렇게 짰냐 / 남이 물으면 뭐라 답하지"처럼 **구조의 설계 의도**를 물을 때 직접 호출한다(채팅 시스템·하네스 둘 다 설명). `change-explainer`(특정 PR의 변화)와 구분: 이쪽은 서 있는 구조의 근거.

## 후속 작업

"다시/재실행/보완/수정/이전 결과 기반" 요청은 Phase 0에서 `_workspace/` 확인 후 해당 에이전트만 부분 재실행한다. 각 에이전트는 이전 산출물이 있으면 읽고 델타만 반영한다.
