# 채팅 시스템 구현 계획

명확해진 요구를 개발자가 바로 착수할 수 있는 계획서로 만든다. 코드는 쓰지 않는다.

## 절차

1. **코드베이스를 읽는다.** 영향받는 파일·함수·데이터 흐름을 실제로 확인한다. 추측 계획 금지.
2. **PR 단위로 쪼갠다.** 한 PR = 한 논리적 변경 = squash 후 커밋 하나 = revert 하나로 롤백. 큰 작업은 독립적으로 리뷰·병합 가능한 단계로 나눈다.
3. **각 단계에 검증을 붙인다.** 무슨 테스트(유닛/경계면/브라우저)로 통과를 판정할지 미리 정한다.
4. **리스크·롤백을 표시한다.** 비가역 지점(데이터, 외부 API, 인메모리 소실)과 대비책.

## 이 프로젝트에서 반드시 고려할 것

- **실시간 vs 내구성 경로**: 실시간 표시(소켓/분석)에 영향 주는 변경은 디스크 쓰기를 백그라운드 큐에 두는 구조를 유지하게 설계.
- **증분 집계**: 데이터가 쌓이는 처리를 매 주기 전체 재계산으로 설계하지 않는다.
- **경계면**: 서버 API ↔ socket.io payload ↔ React 훅. 이 경계를 지나는 데이터 구조를 계획에 명시하고, 세 계층을 함께 바꾸도록 단계를 잡는다(`src/shared/types.ts` 공유 타입 갱신 포함).
- **provider 대칭**: chzzk 기능이면 soop도 필요/가능한지 판단. 공유 로직은 공유 모듈(예: `reconnectBackoff`, `frameSeconds`)로.
- **장수 프로세스**: ffmpeg 등은 시작/정체감지/재시작/종료 시퀀스를 설계에 포함.

## 산출 형식

```
## 목표
[한 문장]

## 영향 범위
- path:line — 무엇을 바꾸나 / 왜

## 단계 (각 = PR 후보)
1. [제목]
   - 변경: ...
   - 검증: [테스트/브라우저 확인]
   - 리스크: ...

## 리스크·롤백
- [비가역 지점] → [대비책 / 되돌리는 법]

## 테스트 계획
- 유닛: ...
- 경계면: 서버↔클라 shape 확인 지점
- 브라우저: 관찰로 검증할 동작
```

## 왜 PR 단위로 쪼개나

이 프로젝트는 롤백 용이성을 위해 "PR 하나 = master 커밋 하나 = revert 하나"를 지킨다. 계획 단계에서부터 이 단위로 나눠야 개발·검수·PR이 그 리듬을 탄다. 한 PR에 여러 논리적 변경을 섞으면 나중에 일부만 되돌릴 수 없다.

산출한 계획서는 `_workspace/`에 저장해 developer가 참조하게 한다.
