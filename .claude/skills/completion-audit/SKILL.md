# 완료 주장 감사

전제 하나: **주장을 믿지 않는다. 직접 확인한다.** 요약·보고·"~한 것 같다"가 아니라 파일·명령 출력·실제 동작이라는 1차 증거로만 통과를 준다.

## 이 프로젝트에서 실제로 터진 거짓 완료 (감사 체크리스트)

- 쓰기가 **거부됐는데** "재시도 성공" → 파일 실존 확인 안 함
- 커맨드가 **push만 실행**됐는데 "커밋 완료" → `git log`로 커밋 존재 확인 안 함
- **재확인 없이** "제대로 됐다" → 근거 없는 주장 반복
- 테스트 **안 돌리고** "통과" → 실제 실행 출력 안 봄
- "됐다"는데 **이미지 안 나옴** → 실제 렌더링 관찰 안 함

공통 원인: **중간 성공을 최종 성공으로 착각** + **확인 절차 건너뛰고 낙관 보고**.

## 주장 유형별 1차 증거

| 주장 | 확인 방법 |
|------|----------|
| 파일 생성/수정 | `Read`/`ls`로 존재 + 내용이 주장과 일치 |
| 커밋 | `git log --oneline` (push 성공 ≠ 커밋 존재) |
| PR 생성 | `gh pr view --json state,baseRefName,files` |
| 테스트 통과 | `pnpm test` **직접 실행** (보고 숫자 불신) |
| 타입 통과 | `pnpm typecheck` 직접 실행 |
| 동작/표시됨 | preview 도구로 렌더링·네트워크·DOM 관찰 |
| 머지됨 | `gh pr view --json state,mergedAt` |
| 삭제/정리됨 | 실제로 사라졌는지 확인 |

## 순서

1. 완료 보고를 **검증 가능한 개별 주장**으로 분해.
2. 각 주장을 위 표대로 1차 증거로 대조.
3. **부분 성공을 완료로 포장한 곳**을 특히 노림 (코드 됐지만 테스트 안 돌림, 브랜치 팠지만 커밋 없음, 파일 썼지만 내용 잘림 등).
4. 판정.

## 판정

```
판정: VERIFIED | FAILED | PARTIAL
확인됨: [주장 → 증거]
반증됨: [주장 → 실제 상태(증거) → 무엇이 빠졌나]
미검증: [확인 못 한 이유]
```
반증이 하나라도 있으면 "완료"라 말하지 않는다. 무엇이 실제로 됐고 안 됐는지 정확히 보고한다.

## 원칙

- **낙관 금지** — "아마 됐을 것"은 미검증.
- **증거 없는 통과 없음** — 확인 못 한 걸 통과로 밀지 않는다.
- **정직 최우선** — 나쁘면 나쁜 대로 보고. 좋게 포장하면 이 감사의 존재 이유가 사라진다.

## 왜 이게 마지막 게이트인가

사용자가 이 프로젝트에서 가장 자주 겪은 문제가 "됐다고 했는데 안 된 것"이다. 개발·검수·PR을 아무리 잘해도, 마지막에 완료 보고가 부정확하면 신뢰가 무너진다. 이 감사는 사용자에게 최종 보고하기 직전, 모든 "완료" 주장을 1차 증거로 한 번 더 때리는 안전망이다.
