# 채팅 시스템 코드 검수

병합 전에 진짜 버그를 잡는다. "존재 확인"이 아니라 경계면 교차 비교 + 실패 시나리오 사고가 핵심이다. 각 모듈 완성 직후 점진적으로 검수한다.

## 검수 우선순위

### 1. 경계면 정합성 (최우선)
이 프로젝트 버그의 대부분이 계층 경계에서 난다. **서버 API 응답/소켓 emit과 이를 소비하는 React 훅·컴포넌트를 동시에 열어** 데이터 shape을 비교한다.
- `src/shared/types.ts`의 타입과 실제 응답/emit이 일치하는가
- 서버가 보내는 필드명·옵셔널 여부와 클라이언트가 읽는 방식이 맞는가
- partial payload 병합(소켓 24개 윈도우 + partial 플래그) 로직이 클라 상태와 정합적인가

### 2. 실패 시나리오
정상 입력 말고 이런 상태에서 깨지는지 상상한다:
- 빈 데이터 / 채팅 없는 구간 / 시청자 0
- provider 오프라인·재연결·방송 종료
- 채팅 폭주(초당 수십 건) / chzzk·soop 동시 연결
- ffmpeg 캡처 공백 / 프레임 부분 존재
- 서버 재시작으로 인메모리 상태 소실

> 실제 과거 버그가 전부 여기서 나왔다: 캡처 공백 프레임 깜빡임, 매초 전체 재계산, 이벤트당 로그 폭주, 참여율 분자·분모 기간 불일치.

### 3. 아키텍처 원칙 위반
- 실시간 경로에 디스크 await
- 매 주기 O(전체) 재계산
- 이벤트당 무제한 부수작업(로그/브로드캐스트) — 스로틀·dedupe 없이
- 지표 분자·분모 기간 불일치

### 4. 코드 품질
함수 >50줄, 파일 >800줄, 깊은 중첩, 삼켜진 에러, 하드코딩, mutation, `console.log` 잔존.

### 5. 테스트
새 순수 로직·경계면·엣지 케이스에 테스트가 있는가. `pnpm test`/`pnpm typecheck`를 직접 돌려 통과 확인(보고만 믿지 않기).

## 판정 형식

```
판정: 통과 | 보류
CRITICAL: [보안/데이터손실/명백한 버그 — 병합 차단]
HIGH: [버그/품질 — 병합 전 수정 권고]
MEDIUM/LOW: [개선 제안]
각 지적: "어떤 입력/상태 → 어떤 잘못된 결과" 재현 시나리오 필수
```
CRITICAL 하나라도 있으면 보류 → developer로. HIGH만이면 조건부 경고.

## 왜 시나리오를 요구하나

"이상하다"는 지적은 개발자가 고칠 수 없다. "soop만 연결됐을 때 providerCounts.chzzk가 undefined라 dominantProvider가 빈 객체를 받아 프레임을 안 띄운다" 같은 구체적 재현 경로가 있어야 수정이 정확해지고, 그게 회귀 테스트로 이어진다.
