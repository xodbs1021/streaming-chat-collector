# 변경 설명

실제 diff를 읽고, 사용자가 이해할 언어로 무엇이·어떻게·왜 바뀌었는지 설명한다. 코드 나열이 아니라 "이 변경이 동작/경험에 무슨 의미인지"가 목적이다.

## 원칙

1. **실제 diff에 근거.** `gh pr diff <n>` / `git show <sha>`로 진짜 바뀐 것만. 계획서·추측이 아니라 코드 변경이 근거.
2. **동작 관점으로 번역.** "FrameCaptureManager에 resolver 주입" → "이제 SOOP 방송도 화면 캡처가 된다". 내부 구현을 사용자 체감 변화로.
3. **왜를 붙인다.** 무엇을 바꿨는지만이 아니라 어떤 문제/요청 때문인지.
4. **기술 수준 맞춤.** 코딩 경험 적으면 용어를 풀거나 비유. 깊이 원하면 파일:라인까지.
5. **롤백 한 줄.** 문제 시 어느 PR/커밋을 revert하면 되는지.

## 출력 형식

```
## 이번 변경 요약 (PR #N: 제목)
한 문단 — 무엇이 좋아졌나

## 무엇을 바꿨나
- [사용자 체감 변화] — (내부적으로: 어떤 파일에서 무엇을)

## 왜 바꿨나
- 어떤 문제/요청 때문에

## 확인 방법 / 영향
- 어디서 이 변화를 볼 수 있나
- 되돌리려면: <PR/커밋>
```

## 큰 diff 다루기

핵심 변경 위주로 요약하고, 부수 변경(포맷·리네임·import 정리)은 "그 외 포맷·정리"로 묶어 한 줄 처리한다. diff를 못 가져오면(PR 접근 실패) 원인을 밝히고 로컬 `git log`/`git show`로 대체한다.

## 왜 동작 관점인가

이 산출물은 팀 내부용이 아니라 **사용자용 최종 보고**다. 사용자는 "어떤 함수가 어떻게 바뀌었나"가 아니라 "내 도구가 이제 뭘 할 수 있고, 뭐가 고쳐졌나"를 알고 싶다. 구현 디테일은 사용자가 더 요청할 때만 파일:라인으로 내려간다.
