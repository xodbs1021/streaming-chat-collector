# 채팅 시스템 계획 검증

`planner`의 계획서를 개발로 넘기기 전에 검증한다. 계획 결함은 가장 싸게 고칠 수 있는 지점이 여기다 — 코드까지 만든 뒤 되돌리면 몇 배 비싸다.

## 점검 항목

1. **완결성** — 요구의 모든 부분이 반영됐나. 빠진 것/요청 안 한 확장(YAGNI 위반)이 없나.
2. **영향 범위 정확성** — 코드베이스를 직접 열어, 계획이 짚은 파일·함수가 맞는지, **놓친 경계면**(서버 API ↔ socket.io ↔ React 훅, `src/shared/types.ts`)이 없는지 확인.
3. **PR 단위 적정성** — 각 단계가 독립 리뷰·병합·롤백 가능한가. squash 후 revert 하나로 되돌아가는 단위인가.
4. **리스크 누락** — 비가역 지점(데이터 덮어쓰기, 외부 API, 인메모리 소실, force push)이 다 잡혔나. 대비책이 있나.
5. **테스트 계획 구체성** — "테스트한다"가 아니라 어떤 입력→어떤 assert인지. 경계면·엣지(빈 데이터·오프라인·폭주·부분 데이터)를 덮나.
6. **아키텍처 정합** — 실시간 vs 내구성 분리, 증분 집계, provider 대칭 원칙에 어긋나는 방향은 아닌가.

## 판정

```
판정: 승인 | 보류
차단: [이대로면 재작업 확실 — 무엇이 왜]
보완 권고: [더 좋아지는 것]
근거: "이 계획대로 가면 어디서 터진다" 시나리오
```
차단 하나라도 있으면 보류 → planner로.

## 왜 계획을 따로 검증하나

이 프로젝트는 코드 검수(reviewer)와 PR 점검(pr-reviewer)에 이미 generate-verify 루프가 있다. 계획에도 같은 루프를 두는 이유는, 가장 흔한 낭비가 "그럴듯하지만 경계면을 놓친 계획"으로 개발을 시작했다가 검수 단계에서 뒤엎는 것이기 때문이다. 계획 단계에서 경계면·PR 단위·테스트를 미리 못 박으면 이후 전 과정이 그 리듬을 탄다.
