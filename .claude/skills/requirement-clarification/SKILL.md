# 요구사항 명료화

명령을 개발로 넘기기 전에 "혼자 판단하면 안 되는 것"을 골라 질문으로 만든다. 목표는 흐름을 끊는 게 아니라, 잘못된 방향으로 개발이 진행돼 나중에 되돌리는 낭비를 막는 것이다.

## 판단 순서

1. **먼저 코드로 답할 수 있나?** 파일 위치·기존 동작·스택 같은 건 Read/Grep으로 직접 확인한다. 확인 가능한 걸 질문으로 올리면 사용자를 귀찮게만 한다.
2. **사용자만 답할 수 있는가?** 제품 의도, 우선순위, 트레이드오프 선택, 되돌리기 어려운 결정만 질문 대상이다.
3. **명확하면 통과.** 질문거리가 없으면 "명확함, 진행 가능"으로 즉시 통과시킨다. 통과가 기본값이다.

## 반드시 질문(또는 확인)해야 하는 것

- **파괴적/비가역 작업**: 데이터 삭제·덮어쓰기, 외부 공개(PR·배포·업로드), 실서버 데이터 조작 — 모호하지 않아도 확인 항목.
- **범위 모호**: "~도 되게 해줘"류. 어디까지? 어느 대상? (예: "이미지" → chzzk만? soop도? 호버/클릭 둘 다?)
- **성공 기준 부재**: "제대로/잘"의 판정 기준이 없을 때.
- **기존과 충돌**: 새 요청이 현재 동작이나 이전 결정과 부딪힐 때.
- **숨은 가정**: 사용자가 사실로 전제하지만 코드상 아닐 수 있는 것.

## 질문 작성 형식

"어떻게 할까요?"(열린 질문)가 아니라 선택지와 추천을 준다:

```
Q: [구체적 질문]
   - A) [옵션] ← 추천 (이유 한 줄)
   - B) [옵션]
   - C) [옵션]
왜 사용자에게 묻나: [코드로 못 정하는 이유 한 줄]
```

## 왜 이렇게 하나

이 프로젝트의 최상위 규칙이 "확실하지 않으면 추측하지 말고 물어본다"이다. 추측으로 개발하면 검수·PR까지 갔다가 "이게 아닌데"로 전부 되돌리게 된다. 앞단에서 3~4개 질문으로 방향을 확정하는 비용이, 잘못 만든 걸 롤백하는 비용보다 항상 싸다.

## 출력

```
판정: 명확 | 확인 필요
(명확) 근거 한 줄 → planner로 진행 권고
(확인 필요) 질문 1~4개 (위 형식) — 실제 사용자 질문은 오케스트레이터가 AskUserQuestion으로 실행
```

질문은 최대 3~4개로 추린다. 한 번에 쏟으면 답하기 어렵다.
