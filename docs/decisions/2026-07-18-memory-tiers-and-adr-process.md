# ADR-0001: 세션 컨텍스트 메모리 3계층 + 결정 기록(ADR) 프로세스

**Date**: 2026-07-18
**Status**: accepted
**Deciders**: 김태윤(계층 설계 제안), Claude(검증·구현)

## Context

하네스 템플릿 개발(2026-07-16~18) 중 두 가지 문제가 드러났다:

1. **캐시 오염** — 진행 로그("Stage N 완료", "PR #32 머지됨")가 항상-로드 파일 메모리에 53줄까지 쌓였다. 매 세션 그걸 전부 읽고 시작하니 컨텍스트 낭비이고, claude-mem이 같은 사건을 자동 녹화하므로 중복(검색 시 에코 N벌)이며, 사용자가 볼 수 없는 위치였다.
2. **결정의 휘발** — "왜 A안을 기각하고 B로 갔나"가 대화에만 존재했다. diff에는 결과만 남고, claude-mem 압축본은 추론의 결이 손실되며, 사후 추출은 토큰 비용이 크다. 블로그·면접에서 사고 과정을 보여주려는 목적에 정면으로 배치된다.

## Decision

컴퓨터 기억 계층을 본뜬 3계층 + 영구 큐레이션 1층으로 역할을 고정한다:

| 계층 | 구현 | 담는 것 |
|---|---|---|
| **캐시** | 파일 메모리(항상 로드) | 규칙·확정 결정만 — 진행 로그 금지 |
| **RAM** | `_workspace/progress.md` | 진행 중 작업 상태. Phase 전환마다 갱신, 완료 시 아카이브(해제) |
| **SSD** | claude-mem(자동 캡처) | 전량 이력 — 검색으로 접근 |
| **영구 큐레이션** | `docs/decisions/` ADR | 중요 결정을 **결정 시점에** Nygard 형식으로 |

ADR 트리거: 대안 기각 후 방향 확정 / 대화로만 확정된 결정 / 사용자가 "기록해 두자". 작성 주체: 파이프라인 안=pr-author(같은 PR), 밖=메인 세션(docs-only PR).

## Alternatives Considered

### 1. 진행 로그를 파일 메모리에 계속 기록 (현상 유지)
- **Pros**: 추가 작업 없음
- **Cons**: 캐시가 매 세션 비대해짐, claude-mem과 중복 녹화로 검색 정밀도 하락
- **Why not**: 항상-로드 층은 극소·안정이어야 한다 — 일지는 캐시의 반대 성질

### 2. 기록 전담 에이전트 신설
- **Pros**: 기록 자동화
- **Cons**: claude-mem 자동 캡처와 정면 중복, 매 대화 상시 오버헤드
- **Why not**: "만들기 전에 찾는다" — 이미 있는 자동 기록과 겹침(YAGNI). 부족한 건 캡처가 아니라 큐레이션이었다

### 3. 나중에 claude-mem에서 추출해 정리
- **Pros**: 지금 노력 0
- **Cons**: 압축 과정에서 기각 사유·트레이드오프의 결 손실, 사후 추출 토큰 비용 큼
- **Why not**: 남기려는 것이 바로 그 결이다 — 결정 시점 기록만이 원형을 보존

## Consequences

### Positive
- 항상-로드 컨텍스트 최소화(실측 53줄→15줄) — 세션 시작 비용 절감
- claude-mem 검색 정밀도 상승(중복 에코 제거) + RAM 갱신 자체가 SSD에 자동 캡처(기록 노력 0)
- 세션이 죽거나 컴팩션돼도 `_workspace/progress.md`로 이어받기
- 사고 과정이 git에 영구 축적 — 블로그·면접 포트폴리오 원료(이 문서가 1호)

### Negative
- 결정마다 ADR 작성 비용 → 트리거 기준으로 남발 방지
- progress.md 갱신 규율 필요 → 오케스트레이터가 Phase 전환마다 강제
